# frontmost 优化执行计划

> 目标：让代码更简洁、对外开源后别人**开箱即用**。
> 下面按优先级分组（P0 最该先做），每项都给出 **问题 / 影响 / 改动 / 涉及文件 / 验证**，可逐条勾选执行。
> 结论先行：**没有发现硬性 Bug**，主要是「开源整洁度、默认缓存策略、采集端数据源、代码组织」四类可优化点。

## P0 · 开源整洁度（必做，改动小、收益大）

这些是「别人 clone 下来会觉得别扭」的点，优先清掉。

### [ ] 0.1 删除作者私有环境变量回退 `IMA_WRITE_SECRET`
- **问题**：`writeSecret()` 里 `return env.FRONTMOST_WRITE_SECRET || env.IMA_WRITE_SECRET;`，`IMA_*` 是作者旧服务（ima.bazinga.ink）的遗留命名。
- **影响**：开源用户看到一个无文档、无来源的 env，困惑且像是死代码。
- **改动**：去掉 `|| env.IMA_WRITE_SECRET`，只保留 `env.FRONTMOST_WRITE_SECRET`。
- **文件**：`src/index.js:208`
- **验证**：`npm test && npm run check:syntax`；搜索全仓库无 `IMA_` 残留。

### [ ] 0.2 把示例配置里的个人 handle 改成中性占位
- **问题**：`wrangler.example.jsonc` 默认 `PUBLIC_USER_ID: "bazinga"`；README 示例 `user="bazinga"`。
- **影响**：默认值是作者 ID，用户容易忘改或误以为有特殊含义。
- **改动**：示例统一改为 `"your-handle"`（或 `"me"`）。README 同步。
- **文件**：`wrangler.example.jsonc`、`README.md`
- **验证**：通读 README，示例自洽。

### [ ] 0.3 补文档：宿主可读的 `data-status` / `data-app-name`
- **问题**：widget 现在会把状态反射到宿主元素（`data-status`、`data-app-name`），但 README 未提。
- **影响**：嵌入方（如做自定义 tooltip / 文案）不知道有这个能力，只能去翻 shadow DOM。
- **改动**：README「Customize」段补一张属性表，说明这两个 data-* 何时出现、取值。
- **文件**：`README.md`、`src/widget-source.js`（注释说明即可）
- **验证**：README 描述与 `update()` 中 `this.dataset.status / appName` 实际行为一致。

## P1 · 性能（默认缓存策略）

### [ ] 1.1 `widget.js` 不应 `no-store`，要可缓存
- **问题**：`/widget.js` 返回头是 `cache-control: no-store`，但它是**几乎不变的静态脚本**。
- **影响**：每个访客每次进页面都重新下载 widget.js，浪费带宽与首屏时间——对所有嵌入方都生效，是最大的一处性能点。
- **改动（推荐 ETag 方案，兼顾更新传播）**：
  - 模块加载时对 `WIDGET_JS` 计算一次弱 ETag（如内容长度+哈希）。
  - 响应改为 `cache-control: public, max-age=300, stale-while-revalidate=86400` + `ETag`，并在请求带 `If-None-Match` 命中时返回 `304`。
  - 这样更新 widget 后最长 5 分钟传播，平时走 304 几乎零成本。
- **文件**：`src/index.js`（`/widget.js` 分支）、可加 `src/http.js` 小工具
- **验证**：
  - `curl -I .../widget.js` 看到 `etag` 与 `max-age`；
  - 带 `If-None-Match` 复请求返回 `304`；
  - `/current` 仍保持 `no-store`（不要误改）。

### [ ] 1.2 图标缓存：`immutable` + 采集端「只传一次」会导致图标永不更新
- **问题**：`getIcon` 返回 `cache-control: public, max-age=31536000, immutable`；采集端 `uploadedIcons[bundleId]` 一旦标记就**永不重传**。
- **影响**：app 换了新图标后，前台徽章永远显示旧图标，且无清理入口。
- **改动（二选一）**：
  - **轻量**：去掉 `immutable`，`max-age` 降到如 `604800`（7 天），让 CDN/浏览器周期性回源；
  - **彻底**：图标 KV key 带内容哈希（`icon:<bundleId>:<sha8>`），采集端按哈希判断是否需要重传（顺带天然解决缓存失效）。
- **文件**：`src/index.js`（`getIcon`/`putIcon`/`iconKey`）、`hammerspoon/init.lua`（`uploadIcon`）
- **验证**：换一个 app 图标后，KV 内容更新、徽章随之刷新。

## P2 · 正确性 / 数据源（采集端）

### [ ] 2.1 在采集端源头过滤 `com.apple.loginwindow`
- **问题**：锁屏时 macOS 把 `loginwindow` 激活为前台，`appWatcher` 会把它当成一次 `switch` 上报，覆盖掉「最后一个真实 app」。
- **影响**：离线/锁屏徽章显示 loginwindow（我们已在 widget 端兜底成「月亮」图标，但**源头数据被污染**：默认徽章会显示 `loginwindow · locked`，且 `name` 字段不再是真实 app）。
- **改动**：`reportCurrentApp` 与 `appWatcher` 里，若 `bundleId == "com.apple.loginwindow"` 则 **跳过 switch 上报**（只让 `lock` 事件改 presence）。源头干净后，widget 端的兜底仍保留做双保险。
- **文件**：`hammerspoon/init.lua:129/152`
- **验证**：锁屏 → `GET /current` 的 `name/bundleId` 仍是上一个真实 app，`status` 为 `locked`/`offline`。

### [ ] 2.2 采集端去掉伪造的 Chrome User-Agent
- **问题**：`config.userAgent` 是一段伪装的 Chrome UA，用来绕过 Cloudflare 挑战。
- **影响**：不诚实、易随 CF 策略失效；README 已要求对该 API 域名关闭 WAF/Bot/Browser Integrity，则伪造 UA 多半冗余。
- **改动**：换成诚实标识，如 `frontmost-collector/1.0 (+https://github.com/<you>/frontmost)`；README 明确写「需为该主机名跳过 CF 挑战」。
- **文件**：`hammerspoon/init.lua:5`、`README.md`
- **验证**：关闭挑战后，`report`/`uploadIcon` 仍返回 2xx。

## P3 · 代码结构 / clean code

### [ ] 3.1 把图标逻辑从 `index.js` 拆到 `src/icons.js`
- **问题**：`index.js`（244 行）同时是 Worker 入口 + 路由 + 图标处理 + 校验工具，职责偏多。
- **改动**：抽出 `putIcon/getIcon/isValidBundleId/hasPngSignature/iconKey/safeDecode/MAX_ICON_BYTES` 到 `src/icons.js`，`index.js` 只留「OPTIONS/路由分发」。
- **收益**：入口一眼看清；校验函数变成**可单测**的纯函数。
- **文件**：新增 `src/icons.js`，瘦身 `src/index.js`
- **验证**：`npm run check:syntax`，路由行为不变。

### [ ] 3.2 复用 `DEFAULT_OFFLINE_AFTER_SECONDS` 常量，消除魔法数 150
- **问题**：`presence.js` 导出了 `DEFAULT_OFFLINE_AFTER_SECONDS=150`，但 `index.js:35` 又硬写了 `150`。
- **改动**：`index.js` 改为 `parsePositiveInt(env.OFFLINE_AFTER_SECONDS, DEFAULT_OFFLINE_AFTER_SECONDS)`。
- **文件**：`src/index.js`
- **验证**：单测通过。

### [ ] 3.3（可选，trade-off）widget 源码从「巨型模板字符串」改为独立文件
- **现状**：widget 是 `export const WIDGET_JS = String.raw\`…\``。优点是零构建、单文件直出；**缺点是内层 JS 无高亮/无 lint/无法单测，手改易错**（本次维护已感受到）。
- **改动选项**：
  - A. 把 widget 写成真实 `src/widget.js`，用 wrangler `rules: [{ type: "Text", globs: ["**/widget.js"] }]` 以文本模块 `import`；
  - B. 保持现状，但在 `dev:widget` 之外加一个 lint（把字符串写盘后 `node --check`，CI 里跑）。
- **建议**：开源前若有精力选 A，长期维护体验明显更好；没精力则至少做 B。
- **文件**：`src/widget-source.js` / `wrangler.example.jsonc` / `package.json` / CI

### [ ] 3.4 给 Hammerspoon 全局变量加注释说明
- **问题**：`appWatcher/heartbeatTimer/caffeinateWatcher` 是全局（无 `local`）。这是 Hammerspoon **防 GC 的正确写法**，但读者易误以为是疏漏。
- **改动**：加一行注释：`-- intentionally global so Hammerspoon doesn't GC the watchers`。
- **文件**：`hammerspoon/init.lua`

### [ ] 3.5 补 `src/http.js` / 校验函数的单测
- **问题**：测试只覆盖 `presence.js`。
- **改动**：为 `parsePositiveInt`、`assertBearer`、（拆分后的）`isValidBundleId`、`hasPngSignature` 各加几条用例。
- **文件**：`test/http.test.js`、`test/icons.test.js`
- **验证**：`npm test` 全绿；CI 自动跑。

## 建议执行顺序与验证

1. **一轮提交 P0 + P1**（整洁度 + 缓存）：收益最高、风险最低。
   - 验证：`npm test && npm run check:syntax`；`curl -I` 检查 `widget.js`/`current`/`icon` 三个端点缓存头符合预期。
2. **再做 P2**（采集端源头修正）：需在真机锁屏/唤醒验证 `/current`。
3. **P3 结构重构 + 测试**：纯内部，单测护航。
4. **P4 文档**：开源前最后润色。
5. P5 视后续需要。

> 每次改动遵循：一处改动 → 一个语义清晰的 commit（Conventional Commits）。改 widget 源码后记得 `npm run deploy` 才会对线上站点生效（widget.js 由 Worker 提供）。
